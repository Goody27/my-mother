```

 お気持ちはとても嬉しいのですが、ご理解いただけますと幸いです。」

（山田くんのSlack・MyMomより）
「断っておいたよ。角が立たないように書いといた。お母さんに任せて。」
```

この時に言うこと:

## デモフォールバック計画

ライブデモは7ホップの連鎖（Slack→API GW→Lambda×3→DynamoDB→Bedrock→SQS→Slack）。
チェーン全体の成功確率 ≈ 70%。フォールバックを3パターン必ず準備する。

### パターンA: ライブデモ（本番）
- 実機で実行。成功すれば最高の説得力。
- 使用条件: 本番接続が安定している、ステージにWi-Fiがある

### パターンB: 録画デモ（第1フォールバック）
- 事前に完全動作した様子を録画しておく（最低2テイク）
- ライブデモ失敗 or Wi-Fi不安定と判断したら即切り替え
- 切り替えセリフ: 「こちらが実際に動作した様子です」（嘘をつかない）
- 録画にはCloudWatchのリアルタイムログも含める

### パターンC: 静的スライド（第2フォールバック）
- スクリーンショット3枚: ①送信されたDM ②断り文 ③CloudWatchログ
- 「この3枚が、AIが1分以内に自律実行した全工程の証拠です」
- 技術的アピールより「なぜこれが革命的か」のメッセージに集中

### 判断フロー

```
当日会場到着 → Wi-Fi接続テスト → Bedrockレイテンシ確認
  ↓ OK               ↓ NG
パターンA           パターンB に切り替え
  ↓ デモ中に失敗
パターンB に即切り替え（「録画も用意しています」と前置きしておく）
```

### トラブル対処表

| 状況 | 対処 |
|------|------|
| EventBridgeが1分以上かかる | 「今AIが考えています。お母さんはゆっくりです」で時間を稼ぐ |
| Slack通知が来ない | CloudWatch Logsを先に見せながら「ログには記録されています」 |
| Bedrockがエラー | 直前のログをスクショしておき「先ほどのデモでは」で代替 |

## モニタリング

### CloudWatchダッシュボード

| ウィジェット | 目的 |
|------------|------|
| Lambda呼び出し数（関数別・24時間） | 稼働確認 |
| Bedrock InvokeAgentトークン数 | コスト監視 |
| SQSキュー深度（メイン + DLQ） | 処理積み残し検知 |
| DynamoDB読み書きキャパシティ | スロットリング検知 |
| エラーレートヒートマップ | 異常検知 |

### アラート設定

| メトリクス | 閾値 | アクション |
|-----------|------|----------|
| Lambda エラーレート | > 5% | CloudWatch Alarm → SNS |
| Bedrockレイテンシ | > 30秒 | CloudWatch Alarm → SNS |
| DLQ メッセージ数 | > 0 | CloudWatch Alarm → 手動確認 |
| 依存度スコア | > 80 かつ 14日継続 | 離脱防止通知を自動送信 |
